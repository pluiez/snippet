# Snippet — 实施任务清单 (TASKS)

> 把 SPEC + ARCHITECTURE 拆成可独立交付的工作单元。每个切片有明确的能力范围与验收标准。
>
> 具体实现细节——文件命名、模块组织、函数签名、IPC 命令命名、具体常量值、UI 实现技巧、特定库的选用——**由 coding agent 在实现时确定**，遵循 ARCHITECTURE 给出的整体架构约束与 SPEC 给出的产品规则。本文档不预设这些决策。

---

## 总体策略

**按垂直切片推进**：每个切片是 UI + 逻辑端到端的完整小功能，可独立演示和验收。不按"先全部 UI 再全部逻辑"的水平分层。

阶段概览：

- **Phase 1 — 脚手架**（2 个切片）
- **Phase 2 — 核心功能**（7 个切片）
- **Phase 3 — 打磨与发布**（3 个工作流）

每个切片包含：**目标**（交付什么能力）、**范围**（属于本切片的功能边界）、**验收**（如何确认完成）、**不包含**（明确划出留给后续切片的内容）。

---

## Phase 1：脚手架

### Slice 0 — 项目初始化

**目标**：跑通 hello-world 级的 Tauri + React 项目，建立工程基础。

**范围**：

- 按 ARCHITECTURE 第 1 节技术栈初始化项目
- 开发环境配好：dev server + 前端热重载 + Rust 自动重建
- 一个最简空主窗口
- 系统托盘图标，点击可显示主窗口
- 应用 single-instance 锁（SPEC 4.10）：第二次启动时通知现有实例前置窗口、新进程立即退出
- 双端类型同步管线打通（Rust 类型可在前端使用对应 TS 定义）
- 日志基础设施配好（控制台输出即可）
- 开发环境启动方法记录在项目根目录文档中

**验收**：

- 启动开发命令后系统托盘出现图标，点击弹出空主窗口
- 改前端代码热重载生效；改 Rust 代码触发重建并自动重启
- 一个示例 Rust 类型能在前端正确 import 对应 TS 类型
- 启动 app 后再次运行启动命令 → 现有实例的窗口被前置，没有新进程产生

**不包含**：任何业务逻辑、任何数据持久化、任何 IPC 业务命令。

---

### Slice 0.5 — 数据层骨架

**目标**：实现数据持久化的基础设施，能从磁盘读出硬编码模板并在主窗口显示一个最简列表。

**范围**：

- 按 SPEC 第 3 节定义所有数据 schema 并在双端同步
- 数据文件夹路径解析逻辑（自定义路径 vs OS 默认）
- 模板的基础 CRUD IPC 命令族
- 启动时扫描数据文件夹并加载到内存索引
- 文件原子写入机制
- `schemaVersion` 字段 + 版本检查（实际迁移逻辑暂用 stub）
- 主窗口以最简形式展示模板 displayName 列表
- 手动放置 sample 模板用于测试

**验收**：

- 手动放置 sample 模板文件，启动 app 后主窗口看到模板名列表
- 通过 IPC 修改模板后文件被持久化，重启 app 状态保持
- 启动时遇到损坏 JSON 文件 → 跳过并在日志中 warning
- `schemaVersion` 不匹配时启动有明确处理（拒绝加载或 migration）

**不包含**：模板编辑 UI、变量系统、tag、颜色、palette、热键。

## Phase 2：核心功能

### Slice 1 — 主窗口模板列表 + 新建 / 复制 / 删除

**目标**：主窗口能完整管理"裸模板"（仅 displayName 和 body，无变量、无 tag、无颜色），覆盖基础 CRUD。

**范围**：

- 主窗口左侧导航：暂仅"全部模板"一项
- 模板列表：每条显示 displayName
- "新建模板"功能：简单编辑面板（改 displayName + body）
- "复制模板"功能：从选中模板创建副本
- "删除模板"功能，带确认
- 编辑面板的保存 / 取消语义
- 数据变更后前端列表自动刷新（通过后端事件机制）

**验收**：

- 能新建、复制、编辑、删除模板，所有操作落盘
- 重启 app 后状态持久
- 修改后列表实时刷新

**不包含**：变量系统、tag、palette、热键、颜色、自动粘贴。

---

### Slice 2 — 变量填充对话框

**目标**：完整实现变量类型系统。从主窗口的"试用"按钮触发填充对话框（暂不接入 palette）。

**范围**：

- 完整实现 SPEC 第 5 节定义的变量类型系统：text / enum、required、剪贴板填充、记住上次值、静态默认值、默认值层叠规则、enum last-used 失效回退
- 变量在模板正文中按 GUID 占位
- 模板渲染（占位符替换）的 IPC 命令族
- 上次使用值的全局持久化（按 display name，跨模板共享）
- 变量填充对话框 UI：左右分栏（左表单 / 右只读实时预览）
- 字段类型对应的输入控件（text 多行 textarea、enum 下拉）
- 字段顺序按 GUID 在 body 中首次出现顺序排列
- 必填校验对提交按钮的影响
- 完成填充后写剪贴板并更新使用记录
- 主窗口模板列表中加"试用"入口触发对话框

**验收**：

- 手动构造一个带 text + enum 变量的模板
- 点击"试用"弹出对话框，填值时预览实时更新
- 提交后剪贴板有正确的渲染结果
- enum 上次使用值在下次打开时回填
- 修改 enum options 后，原上次值如失效则正确回退到下一优先级
- 使用记录字段（lastUsedAt、useCount）在每次完成后更新

**不包含**：编辑模式、palette、颜色 map、自动粘贴。

---

### Slice 3 — 模板编辑模式 + 变量 GUID 体系

**目标**：填充对话框中的"解锁编辑"功能；模板的完整编辑能力；变量 GUID 稳定性契约。

**范围**：

- 编辑模式 UI，与填充模式视觉明显区分（按 SPEC 4.7 节要求）
- 编辑模式可改：displayName、body、变量（增 / 删 / 改名 / 改类型 / 改 options / 改属性）、tag 列表
- body 编辑器：用户面显示 displayName 形式的占位符，存储自动用 GUID
- 变量 GUID 稳定性（SPEC 第 13 节不变量 1、2）
- 剪贴板互斥的 UI 行为（SPEC 5.4 节）
- 编辑模式保存 / 取消的状态保留语义（SPEC 4.7 节往返规则）
- 主窗口的"新建模板"统一走相同的编辑 UI

**验收**：

- 在填充对话框中切到编辑模式
- 改变量 displayName → 切回填充 → 已填值仍在
- 删除变量 → 切回 → 对应字段消失、其他字段保留
- 取消编辑 → 改动丢弃、已填值保持
- 给变量 B 勾选"从剪贴板" → 同模板内 A 的勾自动消失 + 短暂提示
- 视觉上填充模式 vs 编辑模式不可混淆

**不包含**：palette、颜色 map、自动粘贴。

---

### Slice 4 — Palette + 全局热键 + 搜索 + 排序

**目标**：palette 成为 app 的主入口；搜索、排序、窗口互斥逻辑完整。

**范围**：

- Palette 作为独立 Tauri 窗口（无边框、置顶、屏幕居中、按需显示）
- 全局热键注册（默认组合写死，自定义留 Slice 7）
- 热键回调遵守 ARCHITECTURE 第 6 节的关键时序约束（焦点窗口捕获时机）
- 主窗口状态判断：开 → 主窗口前置 + 视觉反馈；关 → 弹 palette
- Palette UI：左右分栏，左搜索 + 列表，右预览
- 完整键盘操作（SPEC 4.2 节键盘操作表）
- 后端实现 SPEC 第 7 节定义的搜索与排序（pinyin 模糊匹配、加权评分、空 / 非空查询不同排序规则）
- 选中分支：零变量直接输出，带变量同窗口变换为填充对话框
- 主窗口 ↔ palette 互斥逻辑完整实现（SPEC 4.9 节）
- Pinning 最小可见行为：模板字段 + 主窗口操作入口 + palette 空查询排序 + 图钉图标

**验收**：

- 任意窗口下按热键都能唤起 palette
- pinyin 全拼、首字母、英文 fuzzy 三种匹配方式都能工作
- 选中零变量模板回车 → 剪贴板有内容 + palette 关闭
- 选中带变量模板回车 → 同窗口变换为填充对话框
- 主窗口打开时按热键 → 主窗口前置 + 视觉反馈，palette 不弹
- palette 打开时唤起主窗口 → palette 关闭
- 置顶模板在 palette 空查询时排在最上

**不包含**：颜色 map（仍 hardcode）、自动粘贴、颜色管理页、热键自定义 UI。

---

### Slice 5 — 颜色系统 + 中央 map + 管理页

**目标**：变量与 tag 的颜色按中央映射查表；管理页可重置 / 自定义；GC 在启动 / 关闭运行。

**范围**：

- 实现 SPEC 第 6 节定义的颜色系统：两个独立中央 map、OKLCh 随机生成、对比度校验、自动 ensure 机制、启动 / 关闭 GC
- 颜色 map 持久化
- 模板保存时自动 ensure 涉及的变量 + tag 颜色
- 主窗口新增"颜色管理"子页（两个 tab：变量颜色 / tag 颜色）
- 管理页操作：单项随机刷新、自定义颜色、"重置全部"、保存 / 取消的暂存模式
- 所有 UI 切换为从颜色 map 查色
- Tag 在主窗口可点击触发筛选；在 palette 中不可点击（SPEC 9.2、9.3 节）
- 颜色变更通过事件机制刷新所有相关 UI

**验收**：

- 新建带新变量名 / tag 的模板 → 颜色 map 自动出现条目
- 主窗口、palette、填充对话框中该名字的颜色一致
- 管理页改色后所有 UI 同步刷新
- 管理页改色后"取消" → 改动丢弃
- 删除某 tag 所有引用后重启 → GC 清除该条目
- tag 在主窗口与 palette 的可点击 / 不可点击区别明显（cursor、hover、点击行为）
- 变量 placeholder 始终无 hover 反馈

**不包含**：明暗主题适配、自动粘贴、完整设置 UI。

---

### Slice 6 — 输出（剪贴板 + 自动粘贴）

**目标**：完整实现输出路径，含 autoPaste 选项与失败降级。

**范围**：

- 实现 SPEC 第 8 节定义的输出路径：剪贴板写入始终发生；autoPaste 开启时焦点切回 + 模拟粘贴
- 严格遵守 ARCHITECTURE 第 6 节的关键时序约束（焦点窗口捕获时机）
- 任一步失败的降级处理：仅剪贴板 + 非阻塞 toast
- 输出完成后正确更新使用记录与上次使用值（与 Slice 2 对接）
- 设置中加 autoPaste 开关（完整设置页 UI 留 Slice 7）

**验收**：

- 默认 autoPaste 关闭：选模板 → 剪贴板有内容
- autoPaste 开启 + 文本编辑器场景：选模板 → 文本自动出现在编辑器中
- 原焦点窗口已关闭的场景：降级 toast 出现 + 剪贴板仍正确
- 每次输出后使用记录正确更新

**不包含**：完整 onboarding、完整设置 UI。

---

### Slice 7 — Onboarding + 完整设置页

**目标**：首次启动体验完整；所有设置项可在 UI 调整。

**范围**：

- Onboarding 窗口与流程，实现 SPEC 第 11 节的三选一
- Onboarding 相关 IPC 命令族（数据文件夹状态检测、初始化、导入与格式校验）
- 完整设置页 UI，覆盖 SPEC 第 12 节列出的所有设置项
- 全局热键的捕获组件 + 冲突校验 + 即时生效
- 主题切换的实现（含 system 模式跟随 OS 偏好）
- 设置变更立即持久化并通知前端

**验收**：

- 干净环境启动 → onboarding 出现
- 三种选项分别走通：默认路径新建、自定义路径新建、从已有路径导入
- 改热键 → 立即生效（按新键能唤起 palette）
- 改主题 → UI 立即切换
- 重启后所有设置保持

**不包含**：主题颜色对比度自动适配。

## Phase 3：打磨与发布

按工作流推进，不再强制顺序，但建议先 A → B → C。

### 工作流 A — 动画与过渡

- palette 唤起的淡入
- palette → 填充对话框的 morph 过渡
- 编辑模式 ↔ 填充模式切换过渡
- 剪贴板互斥的"已从 X 转移"toast 淡出
- 主窗口前置 + 发光描边的脉冲反馈
- 颜色变化的过渡

### 工作流 B — 错误处理与边界态

完整覆盖 ARCHITECTURE 第 8 节的错误处理表，包括但不限于：

- 数据文件夹无写权限的引导
- 单个模板 JSON 损坏的隔离与提示
- 全局热键被占用的提示与处理
- WebView2 缺失的引导安装
- autoPaste 失败的降级 toast
- Onboarding 中导入路径格式不匹配的友好提示
- 所有空状态 UI（palette 无匹配、模板列表无模板、tag 筛选无结果、颜色管理页无条目）

### 工作流 C — 测试与发布

- 单元测试覆盖 SPEC 第 13 节列出的所有核心不变量
- 关键 IPC 命令族的集成测试
- pinyin 索引的多音字回归测试
- fuzzy 匹配评分排序的回归测试
- Windows 安装包打包（按 ARCHITECTURE 第 10 节）
- 代码签名（v1 可选）
- 手动 smoke test：完整安装 → onboarding → 使用流程 → 设置变更 → 卸载 → 重装（数据保留）

## 切片完成的统一标准

每个 Phase 2 切片完成必须满足：

1. 实现该切片"范围"列出的所有能力
2. 通过该切片"验收"列出的所有场景
3. 不破坏已完成的切片（已有切片的验收场景仍通过）
4. 涉及的核心不变量有单元测试覆盖（剩余不变量留 Phase 3 补齐）
5. 双端类型契约同步无遗漏

建议每个切片完成后在 git 上打 tag，便于回滚。
